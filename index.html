四、归因引擎详细设计

4.1 六项核心能力

1

结构归因

从"静态比例"升级为"变化量贡献"

旧: 贡献度 = childValue / parentValue
新: 变化贡献 = (child_now - child_prev) / (parent_now - parent_prev)

使用差异分解法: ΔG = P̄ₙ × ΔQ + Q̄ₒ × ΔP + ΔP × ΔQ

2

维度归因

回答"哪个地区/渠道/终端导致变化最大"

60%
华南 -30万
30%
华北 -15万
15%
西南 +5万
-5%
3

主因路径分析

组合结构归因 + 维度归因 + 异常检测为完整根因链路

LLM 不做数值计算，只做编排和解释。

轻量 Agent 核心实现

public class AttributionAgent {
    private final LlmService llmService;
    private final ToolRegistry toolRegistry;
    private final ConversationManager conversation;

    public Stream<String> analyze(String userQuery, String treeId) {
        String systemPrompt = PromptBuilder.buildSystemPrompt(
            toolRegistry.getToolDescriptions()
        );
        List<Message> messages = conversation.getHistory();
        messages.add(new UserMessage(userQuery));

        for (int i = 0; i < 5; i++) {
            LlmResponse response = llmService.chat(systemPrompt, messages);
            if (response.hasToolCalls()) {
                for (ToolCall call : response.getToolCalls()) {
                    Object result = toolRegistry.execute(
                        call.getName(), call.getParams()
                    );
                    messages.add(new ToolResultMessage(
                        call.getId(), toJson(result)
                    ));
                }
            } else {
                return Stream.of(response.getText());
            }
        }
        return Stream.of("分析完成，请查看归因面板。");
    }
}

八、四套方案对比

方案 AMVP 可选

插件内增强

  • 全部代码在现有插件内
  • Java Agent 复用 LlmService
  • JDK8 + Commons Math + X6
  • 无额外运维
优：开发快、部署简单
劣：归因引擎与插件耦合
方案 B早期不推荐

归因引擎独立 + Agent 独立

  • 3 个独立服务 + 1 个前端插件
  • Java 微服务 + Python Agent
  • Redis + 消息队列
优：解耦清晰
劣：运维高、部署复杂
方案 C有风险

Agent 中心化

  • Agent 为核心，归因做工具
  • 对话为主入口
  • 重度依赖 LLM Function Calling
优：交互体验好
劣：延迟大、成本高、受 LLM 可用性制约
方案 D推荐

混合渐进式

  • MVP = 方案 A，成熟后向 B 演进
  • 确定性分析为主，对话为辅
  • 阶段一插件内 → 阶段二独立 JAR → 阶段三独立服务
优：快速上线、风险可控、渐进演进
劣：初始设计需预留拆分接口
